# 文本向量化（Embedding）实现原理

> 深度解析文本如何转化为 768 维向量，涵盖三种 Provider 的实现方式和 BERT 模型的内部推理过程。

---

## 一、三种 Provider 的向量化实现

`ModelBuilder.getEmbeddingModel()` 根据 provider 创建三种不同的 Embedding 模型：

### Provider 1: Ollama — HTTP 调用本地服务

```java
OllamaEmbeddingModel.builder()
    .baseUrl("http://localhost:11434")
    .modelName("text2vec-base-chinese")
    .build();
// 内部: POST http://localhost:11434/api/embeddings → 返回 float[768]
```

请求/响应流程：
```
embed("如何配置Redis？")
    → HTTP POST /api/embeddings
      Body: {"model":"text2vec-base-chinese","prompt":"如何配置Redis？"}
    → Ollama 服务端运行模型推理
    → Response: {"embedding":[0.012, -0.034, 0.056, ..., 0.078]}
    → 返回 float[768]
```

### Provider 2: OpenAI — HTTP 调用远程 API

```java
OpenAiEmbeddingModel.builder()
    .baseUrl("http://api.openai.com")
    .apiKey("sk-xxx")
    .modelName("text-embedding-ada-002")
    .build();
// 内部: POST /v1/embeddings → 返回 float[1536]
```

请求/响应流程：
```
embed("如何配置Redis？")
    → HTTP POST /v1/embeddings
      Headers: Authorization: Bearer sk-xxx
      Body: {"model":"text-embedding-ada-002","input":"如何配置Redis？"}
    → OpenAI API 服务端推理
    → Response: {"data":[{"embedding":[0.012, -0.034, ...]}]}
    → 返回 float[1536]
```

### Provider 3: Local ONNX — JVM 进程内直接推理

```java
new OnnxEmbeddingModel(
    "saveDir/onnx/model.onnx",       // 神经网络权重文件 (~400MB)
    "saveDir/onnx/tokenizer.json",   // 分词器配置
    PoolingMode.MEAN                  // 池化策略
);
// 内部: ONNX Runtime → CPU 直接推理 → 返回 float[768]
```

推理流程：
```
embed("如何配置Redis？")
    → 加载 ONNX 模型到内存（首次）
    → 分词 → 构建 Input Tensor
    → ONNX Runtime Session.run()
    → 取输出 Tensor → MEAN 池化
    → 返回 float[768]
```

### 统一的调用接口

LangChain4j 抽象了统一的 API，屏蔽了三种 Provider 的差异：

```java
// 单条文本向量化（查询阶段）
Response<Embedding> response = embeddingModel.embed("如何配置Redis？");
float[] vector = response.content().vector();  // float[768]

// 批量向量化（写入阶段）
Response<List<Embedding>> response = embeddingModel.embedAll(textSegments);
List<float[]> vectors = response.content().stream()
        .map(Embedding::vector).collect(Collectors.toList());
```

## 二、三种 Provider 对比

| 维度 | Ollama | OpenAI | Local ONNX |
|------|--------|--------|------------|
| 执行位置 | 本地 HTTP 服务 | 远程 HTTP 服务 | JVM 进程内 |
| 网络开销 | 有（localhost HTTP） | 有（远程 HTTP + TLS） | 无（本地内存计算） |
| 模型文件 | Ollama 服务端管理 | API 服务端管理 | 本地磁盘 model.onnx (~400MB) |
| 推理引擎 | 服务端 GPU/CPU | 服务端 GPU | 本机 CPU（ONNX Runtime） |
| 费用 | 免费 | 按 token 计费 | 免费 |
| 适用场景 | 开发/测试，有 GPU | 生产环境，大规模 | 离线/内网/零延迟需求 |
| 依赖 | Ollama 服务运行中 | 网络可达 + API Key | 模型文件已下载 |

## 三、文本变向量的内部过程

以项目默认模型 `shibing624/text2vec-base-chinese`（768维）为例，基于 BERT 架构。

### Step 1：分词（Tokenization）

```
原始文本: "如何配置Redis连接？"
    ↓ tokenizer.json / 分词器
Token序列: [如, 何, 配, 置, Re, ##dis, 连, 接, ？]
    ↓ 转为整数 ID（词表中的索引）
Input IDs: [1962, 868, 5728, 5309, 2121, 2335, 6834, 2562, 8043]
    ↓ 加上特殊标记
最终输入:  [CLS, 1962, 868, 5728, 5309, 2121, 2335, 6834, 2562, 8043, SEP]
```

每个 token 变成一个整数 ID，这就是神经网络的输入。

### Step 2：词嵌入查找（Embedding Lookup）

```
Input IDs → 查 Embedding 层（词嵌入矩阵，大小 = 词表大小 × 768）
         → 每个 token ID 对应矩阵中的一行
         → 每个 token 变成一个 768 维向量

[CLS]  → [0.02,  0.15, -0.08, ...]
如     → [0.33, -0.11,  0.07, ...]
何     → [0.08,  0.21, -0.15, ...]
配     → [0.19,  0.05, -0.28, ...]
置     → [0.14, -0.09,  0.31, ...]
Re     → [0.45,  0.12,  0.03, ...]
##dis  → [0.37, -0.18,  0.22, ...]
连     → [0.06,  0.27, -0.14, ...]
接     → [0.21, -0.03,  0.19, ...]
？     → [0.01,  0.08,  0.05, ...]
[SEP]  → [0.03, -0.07,  0.11, ...]
```

### Step 3：Transformer 编码（12 层）

```
11个 token 向量（每个768维）
    ↓
┌─────────────────────────────────────────┐
│  Layer 1                                │
│    多头自注意力 (12 heads)               │
│    → 每个 token 关注其他所有 token        │
│    → 捕获上下文关系                      │
│    前馈网络 (FFN)                        │
│    → 非线性变换                          │
│    层归一化 (LayerNorm)                  │
│    残差连接 (Residual)                   │
├─────────────────────────────────────────┤
│  Layer 2 ... Layer 12 (重复相同结构)     │
│    逐层抽象语义信息                      │
└─────────────────────────────────────────┘
    ↓
输出: 11个 token，每个仍然是 768 维
     但每个向量已经融合了整句话的上下文信息
```

**自注意力的作用**：让 "Redis" 这个 token 的向量不仅包含 "Redis" 本身的语义，还融入了 "配置"、"连接" 等上下文信息。

### Step 4：池化（Pooling）— 多 token → 单向量

```
11个 token 向量（每个768维）
    ↓ PoolingMode.MEAN（项目使用的策略）
    ↓ 对所有 token 向量逐元素取平均值

result[0] = mean(0.02, 0.33, 0.08, 0.19, 0.14, 0.45, 0.37, 0.06, 0.21, 0.01, 0.03) = 0.172
result[1] = mean(0.15, -0.11, 0.21, 0.05, -0.09, 0.12, -0.18, 0.27, -0.03, 0.08, -0.07) = 0.036
...
result[767] = ...

最终: 1个 768 维向量 [0.012, -0.034, 0.056, ..., 0.078]
```

**池化策略对比**：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| **MEAN**（本项目） | 所有 token 向量取平均 | 通用句子表示，效果稳定 |
| CLS | 只取 `[CLS]` token 的输出 | BERT 原始设计，微调后效果好 |
| MAX | 所有 token 向量取逐元素最大值 | 突出最显著特征 |

### Step 5：L2 归一化

LangChain4j 的 Embedding 模型通常会对输出做 L2 归一化：

```
原始向量: [0.012, -0.034, 0.056, ..., 0.078]
    ↓ L2归一化: v / ||v||，其中 ||v|| = sqrt(sum(v[i]²))
    ↓
单位向量: [0.003, -0.009, 0.015, ..., 0.021]   // L2 范数 = 1
```

归一化后，余弦相似度 = 向量点积，PgVector 的距离计算更高效。

## 四、为什么写入和查询必须用同一个模型

```
写入时: "Redis连接需要配置host"  ──→ Model_A ──→ [0.01, -0.03, 0.05, ...]
查询时: "如何配置Redis？"        ──→ Model_A ──→ [0.02, -0.02, 0.04, ...]
                                              ↑ 同一个向量空间，距离近 ✓

写入时: "Redis连接需要配置host"  ──→ Model_A ──→ [0.01, -0.03, 0.05, ...]
查询时: "如何配置Redis？"        ──→ Model_B ──→ [0.88,  0.12, -0.91, ...]
                                              ↑ 不同向量空间，距离无意义 ✗
```

每个模型有自己的**词嵌入矩阵和 Transformer 权重参数**，产生的是模型私有的向量空间。不同模型的 768 维向量之间没有可比性 — 第 1 维在 Model_A 里可能表示"技术相关"，在 Model_B 里可能表示完全不同的语义。

### 项目如何保证一致性

```java
// AiChatServiceImpl.getEmbeddingModel()
private EmbeddingModel getEmbeddingModel() {
    // 统一从 sys_config 读取默认 embedding 模型 ID
    String value = sysConfigService.selectConfigByKey("ai.model.embedding");
    if (StringUtils.isBlank(value)) {
        // 如果没配，取数据库中第一个 embedding 模型
        Model first = models.get(0);
        return modelBuilder.getEmbeddingModel(first);
    } else {
        // 按 ID 精确匹配
        long embId = Long.parseLong(value);
        Model model = models.stream().filter(x -> embId == x.getId())
                .findFirst().orElse(null);
        return modelBuilder.getEmbeddingModel(model);
    }
}
```

写入阶段（`KnowledgeBaseController.embedding()`）和查询阶段（`AiChatServiceImpl.chat()`）都通过相同的方式获取 Embedding 模型，确保一致性。

## 五、维度说明

| 模型 | 维度 | 说明 |
|------|------|------|
| text2vec-base-chinese | 768 | 项目默认，中文优化 |
| all-MiniLM-L6-v2 | 384 | langchain4j-embeddings 默认，英文为主 |
| text-embedding-ada-002 | 1536 | OpenAI |
| text-embedding-3-small | 1536 | OpenAI 新版 |
| bge-large-zh | 1024 | 中文，效果更好 |

项目 PgVector 表配置为 768 维，如果更换为不同维度的模型，需要重建向量表或修改 `ai.pgVector.dimension` 配置。
